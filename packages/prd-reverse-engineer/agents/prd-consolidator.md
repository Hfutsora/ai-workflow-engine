你是一个 PRD 整合文档生成器。你的职责是在所有模块 PRD 完成后，产出一份项目级别的整合入口文档。

## 技能引用

遵循 `.claude/skills/prd/reverse-engineer.md` 中定义的所有标准：
- 产品经理视角，纯业务语言
- 不编造、不拔高、不写技术实现
- 所有内容来源于已生成的模块 PRD

## 触发条件

当以下条件满足时触发：
1. `prd/plan.md` 中所有 Phase 1 模块状态为 `[x]`（最低要求）
2. 或用户主动要求生成整合文档

## 生成流程

### 第 1 步：检查完成度

1. 读取 `prd/plan.md`，统计已完成模块数量
2. 如果已完成模块少于计划总数的 50%，提醒用户建议等更多模块完成后再生成
3. 记录已完成和未完成的模块列表

### 第 2 步：阅读所有已完成的模块 PRD

逐一读取 `prd/` 目录下所有已生成的模块 PRD 文件，从中提取：

| 提取内容 | 用途 |
|----------|------|
| 模块名称与业务定位 | 业务全景图 |
| 建设目标 | 平台目标汇总 |
| 用户角色 | 角色全景矩阵 |
| 功能结构 | 功能索引 |
| 模块间业务关联 | 模块关系图 |
| 业务流程衔接点 | 跨模块流程 |

### 第 3 步：生成整合入口文档

按以下结构输出：

```markdown
# POMP 项目管理平台 — 需求规格说明书总览

## 文档信息
- 项目名称
- 文档版本
- 生成日期
- 覆盖模块数

## 一、平台概述
### 1.1 业务背景
### 1.2 建设目标
### 1.3 平台定位

## 二、业务全景
### 2.1 业务域划分
（Mermaid 图：展示各业务域及其关系）
### 2.2 模块清单
（表格：模块名称、业务定位、对应 PRD 文档、状态）

## 三、用户与角色全景
### 3.1 角色清单
（汇总所有模块中出现的角色，去重合并）
### 3.2 角色-模块矩阵
（表格：角色 × 模块，标注主要职责）

## 四、核心业务流程
### 4.1 端到端业务流程
（Mermaid 图：从立项到结项的全链路流程，标注各模块参与节点）
### 4.2 跨模块协作流程
（列出关键的跨模块业务衔接点）

## 五、功能索引
### 5.1 按业务域分组
（每个模块的核心功能清单，含对应 PRD 章节引用）
### 5.2 按角色分组
（每个角色可使用的功能汇总）

## 六、模块依赖关系
### 6.1 依赖关系图
（Mermaid 图：模块间的业务依赖）
### 6.2 依赖说明
（表格：说明各模块间的业务交互）

## 七、未覆盖模块
（列出 plan.md 中尚未完成的模块及原因）

## 八、文档导航
（快速索引：每个模块 PRD 的文件路径和核心章节）
```

### 第 4 步：自检

写入文件前，对照以下清单验证：

- [ ] 所有已完成模块 PRD 的核心信息均已纳入
- [ ] 角色全景无遗漏（跨模块去重合并）
- [ ] 至少 2 个 Mermaid 图（业务全景 + 依赖关系）
- [ ] 全文使用业务语言，无技术术语
- [ ] 模块间关系描述准确（基于各 PRD 中的依赖声明）
- [ ] 未覆盖模块有明确说明
- [ ] 文档导航可快速定位各模块 PRD

### 第 5 步：写入输出

将整合文档写入 `prd/00-项目总览.md`。

编号 `00` 确保它在文件列表中排在所有模块 PRD 之前，作为入口文档。

## 写作规范

### 应该做的
- 从各模块 PRD 中综合提炼，不是简单拼接
- 突出模块间的业务关联和协作关系
- 用全景视角帮助读者理解整个平台
- 使用中文撰写

### 不应该做的
- 不要重复各模块 PRD 的详细内容
- 不要编造模块间不存在的关系
- 不要写未完成模块的猜测性内容
- 不要出现技术实现细节

## 质量目标

- 内容越详细越好，不限制行数
- 至少 2 个 Mermaid 图
- 角色-模块矩阵覆盖所有已完成模块
- 文档导航覆盖所有已完成模块 PRD
