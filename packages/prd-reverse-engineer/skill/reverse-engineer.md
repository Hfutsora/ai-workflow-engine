# 需求规格说明书逆向生成规范

## 前置条件

**必须先执行 mini-wiki skill 生成项目 Wiki 文档**，本 skill 依赖 `.mini-wiki/wiki/` 下的静态分析结果作为主要数据源。

如果 `.mini-wiki/wiki/` 目录不存在或内容过时，请先运行 mini-wiki skill 生成/更新项目文档，再执行 PRD 逆向生成流程。

---

从现有系统的”已实现功能”和”可观察业务行为”逆向梳理需求文档，形成标准化《需求规格说明书》。

本文档约束的是“产出方式”和“写作标准”，不是技术实现说明。
所有输出必须面向产品、业务、设计、测试、实施、交付和管理人员可读可评审。

## 角色定位

你不是开发，也不是架构师。
你是产品经理/需求分析师，你的职责是：

- 识别系统当前解决了什么业务问题
- 识别系统服务了哪些用户和角色
- 识别用户在什么场景下完成什么任务
- 识别系统表现出的业务规则、边界条件、数据范围和约束
- 将这些内容整理成规范、清晰、完整、可评审、可验收的需求规格说明书

你不负责解释技术实现，不负责输出代码设计，不负责描述接口、数据库、类结构、服务调用链等开发细节。

## 核心原则

### 1. 以现有行为为事实依据
需求文档只描述当前系统已经具备、已经上线、已经存在于页面、流程、配置或原型中的功能行为。

### 2. 以产品经理视角表达
文档必须围绕“业务背景、建设目标、用户角色、应用场景、功能规则、页面表现、约束条件、验收要求”展开，不从技术实现展开。

### 3. 不编造、不拔高、不延展
不得擅自补充未来规划、规划中功能、理想状态、推测性能力。
代码或系统中无法明确确认的内容，统一标注为“待确认”。

### 4. 只写业务语言，不写技术语言
输出中禁止出现以下内容：
- 代码文件名
- 类名、方法名
- API 路径
- 请求/响应结构
- 数据库表名、字段名
- 中间件、框架、微服务名
- 技术架构描述
- 调用链、服务链路、部署信息

### 5. 文档要可评审、可落地、可验收
输出不是“介绍文案”，而是正式需求文档，应当支持：
- 需求评审
- 方案理解
- 页面设计
- 测试用例设计
- 上线验收
- 交付存档

## 数据来源优先级

逆向生成需求文档时，按以下优先级理解系统：

| 优先级 | 数据源 | 用途 |
|------|------|------|
| 1 | 源代码（Controller、Entity、Service、枚举、前端页面等） | **唯一事实来源**，所有业务行为的最终判定依据 |
| 2 | mini-wiki（`.mini-wiki/wiki/`） | 代码的结构化索引和参考，帮助快速定位和理解代码，但不是真相本身 |
| 3 | 现有需求文档/规格说明书模板 | 对齐章节结构、语言风格、表格格式、颗粒度 |
| 4 | 已上线系统页面与操作路径 | 识别真实功能、入口、页面结构、按钮行为、提示文案 |
| 5 | 原型图/设计稿/交互稿 | 识别功能布局、信息结构、交互流程 |

**核心原则：代码即真相，wiki 是索引。**

说明：
- mini-wiki 是从项目源代码自动生成的静态分析文档，它的价值是提供结构化的索引和概览，帮助快速理解模块划分、关键文件位置、数据模型关系等
- 当 wiki 描述与代码实际行为不一致时，**必须以代码为准**
- wiki 适合作为分析的起点和导航，但具体的业务规则、状态流转、字段定义、校验逻辑等必须从代码中确认
- 代码中提取的信息只能用业务语言转化后写入文档正文，不允许直接出现代码片段、类名、方法名等技术细节

## 适用场景

本 skill 适用于以下任务：

- 从已有系统逆向补全文档
- 从老系统整理正式需求规格说明书
- 为重构项目先补齐当前版本需求基线
- 在交接、验收、审计前统一沉淀需求文档
- 将散落在页面、原型、流程、历史说明中的信息整理成标准需求文档
- 用作新版本改造前的“现状需求基线”

## 逆向分析工作流

## Step 1：确定文档对象和边界

先明确本次文档要描述的是：

- 整个系统
- 一个业务域
- 一个一级模块
- 一个专题需求集合
- 一个版本范围内的需求集合

必须先界定边界，再开始分析。
边界不清时，不得直接开始写文档。

输出时应先明确：
- 本文档对应的项目/模块名称
- 本文档覆盖的业务范围
- 本文档不覆盖的业务范围
- 本文档对应的版本范围或阶段范围（如适用）

## Step 2：识别业务背景与建设目标

从现有系统和历史文档中提炼：

- 为什么会建设这个模块/系统
- 这个系统主要解决什么问题
- 主要服务哪类业务
- 当前版本聚焦哪些建设目标
- 当前版本明确不包含什么内容

这一部分必须写成业务语言，不得写成”系统实现了什么组件”。

### 强制输出「现状 – 痛点 – 价值」（核心流程模块必须）

对核心流程类模块（如合同管理、项目立项、预算管理、外协管理、采购管理、记账中心等），**必须**在概述章节中输出以下三个小节，不得用两三句通用背景代替：

#### 2.1.1 业务现状

描述当前业务怎么做，涉及哪些系统、角色、关键步骤。示例写法：

> 当前教育项目的合同立项流程涉及 DPMS、POMP、CRM 三个系统。项目经理在 CRM 中查看合同信息，在 DPMS 中手动录入项目基本信息，再到 POMP 中关联合同与项目。整个流程需要跨系统操作，信息需要人工同步。

#### 2.1.2 痛点分析

按角色分别列出具体痛点，每个角色 2–3 条，必须写到具体动作和后果层面：

| 角色 | 痛点 |
|------|------|
| 项目经理 | 1. 需要跨多个系统手动维护信息，容易遗漏和不一致 2. 合同变更后需手动核对和更新项目信息 3. 缺少统一视图了解合同与项目关联状态 |
| 管理部门 | 1. 缺乏统一规范导致不同项目经理操作方式不一致 2. 无法实时获取合同到项目的转化进度 |
| IT 部门 | 1. 多系统数据不一致导致频繁排查问题 2. 缺少标准化接口增加维护成本 |

#### 2.1.3 业务价值分析

面向一线用户和管理侧分别说明价值，各 2–3 条：

| 维度 | 价值 |
|------|------|
| 一线效率 | 1. 合同信息自动同步，减少手动录入 2. 一个平台完成从合同到项目的全流程 |
| 管理规范 | 1. 统一合同处理流程，消除操作差异 2. 合同到项目转化全程可追溯 |
| 系统联动 | 1. 与 CRM/OA/OTC 自动集成，消除信息孤岛 2. 与财务系统自动对接，减少人工干预 |

#### 2.1.4 关键干系人

标出需求决策方、关键访谈用户、IT 侧关键用户及其核心诉求。如果是逆向生成，可基于系统角色和代码行为推断，但必须标注”当前依据系统行为推测，需业务确认”：

| 类型 | 角色 | 核心诉求 |
|------|------|----------|
| 需求决策方 | XX 部门负责人 | [待确认] |
| 关键访谈用户 | 项目经理、合同管理员 | 减少跨系统操作、统一流程 |
| IT 侧关键用户 | 系统管理员 | 降低多系统集成维护成本 |

> 以上角色和诉求为当前依据系统行为推测的角色，需业务确认。

## Step 3：识别功能结构

从导航、菜单、首页入口、工作台入口、详情页入口、配置页入口等观察，提炼系统的功能结构。

输出应体现：
- 一级功能模块
- 二级功能模块
- 主要页面或页面组
- 主要任务流转路径

功能结构强调“业务功能组织方式”，不是菜单树照抄，也不是页面清单罗列。

## Step 4：识别用户与角色

从实际使用方式、页面权限差异、操作职责差异中识别角色。

要求：
- 角色名称必须使用业务语言
- 每个角色必须说明“职责”和“在系统中的主要任务”
- 区分“谁发起”“谁审批”“谁执行”“谁查看”“谁维护”
- 如果角色边界不清晰，可先按现有系统观察结果写出，再标记“待确认”

## Step 5：识别业务流程与应用场景

围绕典型任务进行逆向，而不是围绕技术模块逆向。

每个核心功能都应识别：
- 什么场景下会使用
- 谁来使用
- 用户从哪里进入
- 需要完成哪些动作
- 系统给出什么结果
- 遇到异常时系统如何反馈

优先写：
- 主流程
- 关键分支流程
- 典型异常流程
- 与其他模块衔接的流程

## Step 6：识别业务规则与边界条件

重点提炼以下内容：

- 哪些信息必须填写
- 哪些情况下可新增、编辑、删除、提交、撤回、作废、变更
- 哪些操作受角色、状态、组织、项目类型、数据范围等条件约束
- 哪些业务对象存在前后依赖关系
- 哪些提示语、校验语、限制语具有明确业务含义
- 哪些范围仅对特定业务生效

规则必须用自然语言表达，禁止出现技术实现术语。

### 关键规则必须拆成「条件 + 处理逻辑 + 提示文案」

对关键业务规则（例如项目名称生成、概算覆盖利润中心、设备物料过滤、阶段校验、金额校验等），**每条规则必须以以下三段式结构输出**：

1. **触发条件**：什么时候会触发这条规则（用户操作、数据状态、字段值等）
2. **处理逻辑**：系统具体怎么做，必要时用"公式样式"的业务语言
3. **提示文案**：如果校验失败或需要提示用户，用户会看到什么文案

**不允许只写一句"系统会校验 XXX 并提示用户修正"。**

示例——项目名称命名规则：

| 项 | 内容 |
|----|------|
| 触发条件 | 合同生成项目时，系统自动生成项目名称 |
| 处理逻辑 | 项目名称默认取合同名称；用户可手动编辑；名称不能为空；长度上限 200 字符；项目编码在提交审批通过后由系统自动生成 |
| 提示文案 | "项目名称不能为空" / "项目名称超过 200 字符，请缩短" |

示例——概算覆盖利润中心规则：

| 项 | 内容 |
|----|------|
| 触发条件 | 用户拉取概算数据后 |
| 处理逻辑 | 用概算中的利润中心覆盖项目基本信息中的利润中心，覆盖后不允许手动编辑；设备物料的利润中心跟随概算利润中心 |
| 提示文案 | "利润中心已由概算自动填充，不可手动修改" |

### 业务规则必须展开为「可执行描述」

当系统存在"规则引擎""匹配规则""差额计算""多场景分支""策略模式""状态机流转""公式计算"等逻辑时，**禁止只写笼统描述**。

以下写法属于不合格，必须展开：

- "系统根据场景编码自动匹配处理规则"
- "未匹配到特定规则时使用通用记账规则"
- "同一单据重复入账时，系统计算新旧金额差值并更新差额部分"
- "系统自动计算利润率"
- "根据项目类型走不同审批流程"

必须展开为以下颗粒度：

#### 1. 列清楚判断条件

使用表格或列表写出：输入条件 → 触发的规则名称 → 处理结果。

示例：

| 场景编码 | 业务来源 | 费用类型 | 触发规则 | 处理方式 |
|---------|---------|---------|---------|---------|
| SCENE_001 | 合同付款 | 在途费用 | 合同在途记账规则 | 新增在途金额记录 |
| SCENE_002 | 报销入账 | 实际费用 | 报销实际记账规则 | 新增实际金额记录 |
| 未匹配 | 任意 | 任意 | 通用记账规则 | 按默认科目映射记账 |

#### 2. 写出具体处理逻辑

- **金额类**：给出明确计算方式，用接近公式的中文表达：
  - 当前在途金额 = 原在途金额 + 本次记账金额
  - 业务利润 = 合同收入 − 业务成本合计
  - 利润率 = 业务利润 ÷ 合同收入 × 100%
- **状态类**：写明从什么状态变成什么状态，在什么条件下变更：
  - 当所有子项目结项时，母项目状态从"执行中"变为"待结项审批"
  - 当审批通过时，采购单状态从"审批中"变为"已批准"

#### 3. 必要时给出「公式样式」描述

用中文 + 简单公式表达，不写成代码：

- 新在途金额 = 原在途金额 + 本次在途金额
- 新实际金额 = 原实际金额 + 本次实际金额
- 差额 = 本次金额 − 已记账金额
- OTC 利润率 = (合同金额 − OTC 成本合计) ÷ 合同金额 × 100%

#### 4. 对重复入账/差额更新类场景，必须至少回答三个问题

1. **如何识别"同一单据"？** — 通过哪些字段组合判定唯一性（如：业务单据编号 + 费用类型 + 科目编码）
2. **差额如何计算？** — 差额 = 本次提交金额 − 上次已记账金额
3. **差额落在哪些字段/维度上？** — 如：仅更新在途金额，还是同时更新实际金额和累计金额；是否影响利润计算

#### 5. 对多场景分支/策略模式，必须穷举分支

当系统根据项目类型、业务来源、组织类型等走不同处理路径时，必须用表格或决策树列出所有已实现的分支，包括默认/兜底分支。

#### 6. 无法确定时的标准写法

如从现有系统行为或代码中**确实无法确定具体规则**，不能用一句话模糊带过，应写成：

> 当前系统表现为"差额更新"行为，但具体公式和分摊逻辑在现有实现中无法完全确定，需与财务/产品确认后补充。[待确认]

并补充已知信息：
- 已知：系统会对比新旧金额
- 已知：差额会落在 XX 字段
- 未知：差额是否按比例分摊到子科目
- 未知：负差额是否允许（即本次金额小于已记账金额）

## Step 6.5：识别「状态 × 操作」决策表

当模块存在多个状态维度（如立项状态、审批状态、交底状态）且不同状态组合下允许的操作不同时，**必须产出一张「状态 × 操作决策表」**，不允许只写"不同状态展示不同按钮"这种笼统句子。

### 决策表标准格式

| 立项状态 | 审批状态 | 交底状态 | 允许的操作 | 特殊展示规则 |
|----------|----------|----------|-----------|-------------|
| 未立项 | 未发起 | -- | 立项、查看 | -- |
| 合同立项 | 审批中 | -- | 查看 | 列表标记"审批中" |
| 合同立项 | 被驳回 | -- | 编辑、重新提交、查看 | **行高亮显示** |
| 合同立项 | 审批通过 | 未交底 | 查看、发起交底 | 审批通过后列表保留两周 |
| 合同立项 | 审批通过 | 已交底 | 查看 | -- |

### 决策表必须包含

1. **所有关键状态维度**：列出构成组合的每个状态字段
2. **每种组合下允许的操作**：精确到按钮级别（立项、编辑、查看、提交审批、作废、转让等）
3. **特殊展示规则**：
   - 哪些状态组合下需要行高亮或特殊标记
   - 数据保留策略（如审批通过后列表只保留两周）
   - 操作入口的展示/隐藏规则
4. **兜底行为**：未列出的状态组合默认允许什么操作

### 何时必须产出决策表

- 模块有 ≥2 个状态维度，且不同组合下按钮/操作存在差异
- 模块存在"流程类"功能（立项审批、变更审批、交底确认等）
- 模块的列表页上按钮根据状态动态显示/隐藏

## Step 6.6：识别列表视图与页签规则

当模块有多个列表视图或多个页签（如按来源区分、按产品类型区分、按状态区分）时，**必须单独写一段「视图与筛选规则」**。

### 视图规则标准结构

#### 页签定义

| 页签名称 | 筛选条件 | 统计数字含义 | 数字为 0 时 |
|----------|----------|-------------|------------|
| 标品 | 产品类型 = 标准产品 | 统计"未发起"状态的数量 | 不展示该页签 |
| 小综解/大G | 产品类型 ∈ {小综解, 大G} | 统计"待处理"状态的数量 | 仍展示，数字显示 0 |

#### 列表展示规则

| 规则类型 | 具体规则 |
|----------|----------|
| 默认排序 | 按创建时间倒序；被驳回状态置顶 |
| 行高亮 | 被驳回状态行以红色/橙色高亮 |
| 数据保留 | 审批通过的数据在列表中保留两周后转入历史 |
| 列表操作栏 | 根据状态组合动态展示可用操作按钮（参见状态×操作决策表） |

### 何时必须写视图规则

- 模块列表有 ≥2 个页签或视图切换
- 页签上显示统计数字
- 列表存在条件性高亮、排序优先级或数据保留策略

## Step 7：识别数据范围与口径

如果系统存在明显的数据可见范围、业务范围、组织范围、项目范围、版本范围、标签范围等，需要在“需求分析”中单独描述。

应重点关注：
- 谁能看什么
- 哪些数据被纳入当前范围
- 哪些数据被排除
- 特殊项目、特殊组织、特殊标签、特殊流程如何处理

这一部分允许出现业务对象名称，但不允许出现底层取数逻辑描述。

## Step 8：整理功能需求清单

功能需求分析部分应按业务主题组织，不按技术模块组织。

推荐组织方式：
- 开发中需求
- 待做需求
- 已完成优化及补充需求

每个需求点至少应包含：
- 需求标题
- 需求背景/现状问题
- 应用场景
- 功能说明
- 业务规则
- 页面或操作要求（如适用）
- 提示文案或校验要求（如适用）

### 2. 概述

#### 4.2 面向用户

#### 4.3 角色权限

#### 5.3 已完成优化及补充需求
整理已完成但需要纳入规格文档统一管理的优化项、补充项、修订项。

## 单条需求的标准写法

在“5. 功能需求分析”下，每条需求建议采用如下写法：

### 5.x.x 需求标题

#### 需求
说明当前存在的问题、触发背景、业务诉求、调整原因。

#### 场景
用表格列出前置条件、适用范围、备注说明等，例如：
- 平台范围
- 组织范围
- 项目类型
- 用户角色
- 使用条件
- 特殊说明

#### 功能需求 / 功能说明
说明系统应如何支持该需求，重点写：
- 用户能做什么
- 系统会如何表现
- 页面上会发生什么变化
- 哪些信息需要展示
- 哪些信息需要录入
- 哪些操作需要校验
- 哪些结果需要提示或联动

#### 业务规则
当需求涉及复杂约束时，单独列出规则，例如：
- 状态限制
- 角色限制
- 范围限制
- 先后顺序限制
- 必填逻辑
- 互斥逻辑
- 默认逻辑
- 特殊组织/项目差异化规则

#### 提示与校验
当需求涉及明显提示文案、警示语、拦截语、校验语时，需要单独写明。

#### 补充说明
包括但不限于：
- 历史逻辑兼容
- 与其他模块关系
- 与原有流程差异
- 待确认点

## 写作风格要求

### 1. 保持正式文档风格
语言风格应接近企业正式需求规格说明书，避免口语化、聊天式表达。

### 2. 以业务语言表达
优先使用：
- 项目经理
- 项目成员
- 管理员
- 项目范围
- 数据范围
- 提交审批
- 计划执行
- 变更处理
- 预算编辑
- 报工报销
- 收入节点
- 绑定规则
- 校验提示

不得使用：
- 接口调用
- 服务编排
- 字段映射
- 表结构
- DTO
- 入参出参
- 数据落库
- 微服务
- 中间件
- 状态码
- 脚本逻辑

### 3. 章节语言要“像需求文档”，不是“像总结报告”
不要写成复盘报告、调研报告、系统介绍、方案汇报或开发说明。

### 4. 颗粒度要统一
同一篇文档内，章节颗粒度应一致。
不要某些章节写一句话，某些章节堆十页细节。
内容应完整，但不要过度发散。

## 禁止事项

严禁以下行为：

- 编造系统中不存在的功能
- 根据经验脑补业务流程
- 输出未来规划、版本规划、愿景内容
- 将技术实现细节写入需求文档
- 用代码结构替代业务结构
- 直接复制杂乱原文，不做整理
- 只罗列页面，不说明业务意义
- 只罗列规则，不说明适用场景
- 只写主流程，不写关键异常和边界
- 用“可能”“应该”“大概”“推测”这类模糊词代替确认结论

如无法确认，统一写法为：
- 待确认
- 当前系统行为无法明确判断，待业务确认
- 当前资料未体现明确规则，待进一步确认

## 质量标准

每份《需求规格说明书》至少满足以下标准：

### 必要条件
- 严格按模板章节输出
- 文档信息、修订记录、目录齐全
- 引言、概述、原型地址、需求分析、功能需求分析全部完整
- 文档语言统一为产品经理/需求分析师视角
- 不出现代码、API、数据库、技术架构等实现细节
- 每个核心需求项都包含背景、场景、功能说明
- 用户、角色、范围、规则、边界写清楚
- 文档内容支持评审、测试、验收使用
- 对不确定内容有明确“待确认”标识
- 章节编号、标题层级、表格风格一致

### 推荐条件
- 功能结构清晰，便于理解整体范围
- 面向用户和角色权限部分足够完整
- 数据范围/业务边界写得具体可执行
- 特殊业务说明能覆盖历史兼容、特殊项目、特殊组织差异
- 功能需求分析中的每个需求点都能单独评审
- 核心流程模块有完整的"业务现状 → 痛点分析 → 业务价值分析"三段式开头
- 标出了需求决策方、关键访谈用户、IT 侧关键用户及其诉求
- 多状态控制的流程有完整的"状态 × 操作决策表"
- 多视图/多页签的列表有完整的"视图与筛选规则"（筛选条件、数字含义、排序、高亮、保留策略）
- 关键业务规则拆成"触发条件 + 处理逻辑 + 提示文案"三段式，无一句话带过的笼统规则

## 评审标准

prd-reviewer 在审查文档时，重点检查以下问题：

### 结构完整性
- 是否完全按模板写
- 是否漏章节、漏小节、漏关键表格

### 产品视角一致性
- 是否始终站在业务/产品视角
- 是否混入技术实现语言

### 业务背景厚度（核心流程模块）
- 是否存在"业务现状、痛点分析、业务价值分析"三个小节？如果没有，判定为结构不完整
- 是否标出了需求决策方、关键访谈用户、IT 侧关键用户？如缺失，提示补全
- 痛点是否按角色分列、写到具体动作和后果？还是只有两三句通用描述

### 需求完整性
- 是否写清楚背景、场景、对象、动作、规则、结果
- 是否写清楚谁能用、在哪用、何时用、受什么限制

### 状态决策表与视图规则
- 对于有多状态控制的流程，是否给出了"状态 × 操作决策表"？还是只有一张抽象状态图
- 列表/页签是否写清了：筛选条件、数字含义、排序规则、数据保留策略、行高亮规则

### 规则颗粒度
- 关键业务规则是否拆成"触发条件 + 处理逻辑 + 提示文案"三段式？
- 是否存在"系统会校验 XXX 并提示用户修正"这类一句话带过的笼统描述？如果有，要求展开

### 边界清晰度
- 是否说明适用范围与不适用范围
- 是否说明特殊情况和例外场景

### 可验收性
- 是否能让测试据此设计用例
- 是否能让研发和设计理解要做什么
- 是否能让业务评审是否满足诉求

### 规范性
- 标题是否规范
- 编号是否连续
- 表格是否统一
- 文风是否正式
- 是否存在模糊表达

## 模块划分原则

在拆分文档时，遵循以下原则：

1. 按业务域划分  
一个文档应对应一个清晰的业务域、专题域或版本需求集合。

2. 粒度适中  
既不能过粗导致无法评审，也不能过细导致文档碎片化。

3. 与用户认知一致  
优先按用户理解的业务模块划分，而不是按内部实现边界划分。

4. 横切能力可独立成文  
例如权限、通知、规则引擎、数据范围等，可按需要单独形成专题需求文档。

5. 同一文档内要保证上下文自洽  
一个文档中的术语、角色、范围、规则必须前后一致。

## 交叉引用规范

文档中如需引用其他文档或其他章节，统一使用正式写法：

- 参见《XX需求规格说明书》
- 详见 4.4 数据范围/业务边界
- 详见 5.1.3 XX需求
- 相关内容参见《XX模块需求规格说明书》

禁止使用技术引用方式或路径式表达。

## 计划驱动的工作流

PRD 逆向生成采用**计划先行、逐模块生成、可中断恢复**的工作模式。

### 工作流概览

```
第 0 步: 确认 .mini-wiki/wiki/ 已生成且为最新
         ↓
第 1 步: 运行 prd-analyzer → 输出 prd/plan.md（模块计划）
         ↓
第 2 步: 用户确认/调整模块划分和优先级
         ↓
第 3 步: 运行 prd-generator 生成模块 PRD
         支持逐个生成或并行批量生成（后台子 agent）
         每完成一个，更新 plan.md 中对应模块状态为 [x]
         ↓
第 4 步: 阶段性运行 prd-reviewer 审查已生成的 PRD
         ↓
第 5 步: 根据审查报告修正问题
         ↓
第 6 步: 所有模块完成后，运行 prd-consolidator
         生成 prd/00-项目总览.md（项目级整合入口文档）
```

### 计划文件（prd/plan.md）

计划文件是整个流程的核心追踪文件，必须满足：

1. **状态可追踪**：每个模块有明确的完成状态标记
   - `[x]` 已完成
   - `[ ]` 待生成（数据源就绪）
   - `[~]` 需先补充 mini-wiki 再生成

2. **数据源自包含**：每个模块列出完整的 mini-wiki 路径、代码路径和前端路径，使得生成时可以独立启动，无需额外查找

3. **分阶段组织**：
   - Phase 1: wiki 覆盖充分，可直接生成
   - Phase 2: wiki 覆盖不足，需先补充
   - Phase 3: 可选（横切关注点等）

4. **增量更新**：每次生成完一个模块后，只更新该模块的状态标记，不影响其他内容

### 逐模块生成

支持两种执行模式：

#### 模式一：逐个生成（默认）

每次生成一个模块：
1. 从 `prd/plan.md` 读取该模块的数据源列表
2. 按本 skill 的分析工作流和模板要求生成 PRD
3. 输出到 `prd/模块名.md`
4. 将 plan.md 中该模块的状态从 `[ ]` 更新为 `[x]`，并记录完成日期

#### 模式二：并行批量生成（推荐）

将多个模块作为独立的后台子 agent 并行启动，大幅提升效率。

**启动方式：**
- 用户说"启动所有模块"或"并行生成"时，为每个待生成模块启动一个后台 Agent
- 每个 Agent 接收完整的上下文：skill 规范路径、该模块的全部数据源列表、输出文件路径、质量要求
- 所有 Agent 使用 `run_in_background: true` 并行执行

**每个子 Agent 的标准 prompt 结构：**
```
1. 角色声明："你是 PRD 生成器"
2. 规范引用：指向 .claude/skills/prd/reverse-engineer.md
3. 核心约束：产品经理视角、纯业务语言、禁止技术术语、不编造
4. 数据源清单：该模块的全部 mini-wiki 路径 + 辅助代码路径
5. 输出要求：文件路径、Mermaid 图要求、特殊注意事项（内容越详细越好，不限制行数）
6. 完成动作：更新 prd/plan.md 中该模块状态为 [x]
```

**并行安全性：**
- 每个模块的 PRD 输出到独立文件（`prd/模块名.md`），互不冲突
- `prd/plan.md` 的状态更新为各模块独立行，并发写入不会产生逻辑冲突
- 模块间无生成依赖（依赖关系仅影响内容引用，不影响生成顺序）

**进度追踪：**
- 每个子 Agent 完成后会自动通知
- 主 agent 汇总完成情况并向用户报告
- 随时可查看 `prd/plan.md` 了解整体进度

**使用示例：**
```
# 启动全部待生成模块
"启动所有模块的 PRD 并行生成"

# 启动指定批次
"并行生成 母子项目、外部集成、记账中心"

# 查看进度
"当前有哪些模块完成了？"
```

### 中断与恢复

- 中断后随时可以查看 `prd/plan.md` 了解当前进度
- 恢复时只需说"继续 plan.md 中下一个待生成的模块"
- 每个模块独立生成，不依赖前一个模块的生成结果（除非有依赖关系标注）

### 配合的 Agent

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| prd-analyzer | 分析项目结构，生成计划 | 项目代码 + mini-wiki | `prd/plan.md` |
| prd-generator | 按计划逐模块生成 PRD | plan.md 中的模块信息 | `prd/模块名.md` |
| prd-reviewer | 审查已生成 PRD 的质量 | `prd/*.md` | `prd/review-report.md` |
| prd-consolidator | 生成项目级整合入口文档 | 所有已完成的 `prd/*.md` | `prd/00-项目总览.md` |
| prd-updater | 代码变更后增量更新 PRD | git diff + plan.md + 现有 PRD | 更新后的 `prd/模块名.md` + `prd/update-report.md` |

## 增量更新

当项目代码发生变更后，无需重新生成整个 PRD，只需增量更新受影响的模块文档。

### 触发时机

- 代码提交后发现 PRD 与当前实现不一致
- 完成一轮功能开发后需要同步更新需求文档
- mini-wiki 更新后需要同步更新 PRD

### 更新流程

```
第 1 步: 识别变更（git diff / 用户指定）
         ↓
第 2 步: 映射变更文件到 PRD 模块（通过 plan.md 的代码路径）
         ↓
第 3 步: 分析变更的业务含义（新增/修改/删除/无影响）
         ↓
第 4 步: 精确更新受影响的 PRD 章节（Edit，不重写整个文件）
         ↓
第 5 步: 更新修订记录
         ↓
第 6 步: 输出更新报告 prd/update-report.md
```

### 核心原则

1. **最小变更**：只修改受代码变更影响的部分，不重写整个 PRD
2. **精确定位**：通过 plan.md 中的代码路径映射，准确找到受影响的模块
3. **安全删除**：功能删除类变更建议用户确认后再移除 PRD 内容
4. **可追溯**：每次更新都记录在 PRD 的修订记录中

### 配合的 Agent

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| prd-updater | 代码变更后增量更新 PRD | git diff + plan.md + 现有 PRD | 更新后的 `prd/模块名.md` + `prd/update-report.md` |

## 项目级整合文档

当所有模块（或至少 Phase 1 全部模块）PRD 生成完成后，需生成一份项目级别的整合入口文档 `prd/00-项目总览.md`。

### 目的

- 为整个需求规格说明书体系提供统一入口
- 展示平台业务全景、模块关系和角色全景
- 帮助读者快速理解平台整体能力并定位各模块文档

### 触发时机

- Phase 1 所有模块 PRD 生成完成后，可先生成初版
- 后续模块完成后，增量更新

### 生成方式

使用 `prd-consolidator` agent，它会：
1. 读取 `prd/plan.md` 检查完成度
2. 阅读所有已完成的模块 PRD
3. 综合提炼业务全景、角色全景、跨模块流程、功能索引
4. 输出 `prd/00-项目总览.md`

### 配合的 Agent（整合）

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| prd-consolidator | 生成项目级整合入口文档 | 所有已完成的 `prd/*.md` | `prd/00-项目总览.md` |

## 输出要求总结

最终输出的需求文档必须满足以下总要求：

- 结构严格对齐《交付项目管理-需求规格说明书》模板
- 表达严格站在产品经理/需求分析师视角
- 内容只描述业务、角色、场景、规则、页面行为和验收关注点
- 不出现任何技术实现细节
- 文档可直接用于需求评审、设计协作、测试准备和交付沉淀
- 对现状负责，对事实负责，对边界负责，不编造、不拔高、不偷塞需求
